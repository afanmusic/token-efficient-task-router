## Skill 简介

`token-efficient-task-router` 是一个面向 WorkBuddy、iMA Copilot、OpenClaw、QClaw、CodeBuddy 及其他 Agent Skill 系统的标准化控制型 Skill。

它不是业务技能，而是一个：

- Token 成本控制器
- 任务模式路由器
- 执行前安全闸门
- 智能体 ROI 管理器

它的目标是让智能体在真正执行前，先判断任务是否明确、是否复杂、是否高风险、是否值得消耗更多 token，然后再进入合适的工作模式。

## 适用场景

适合以下情况：

- 用户需求模糊，需要先判断该问清楚、先规划还是直接做
- 任务可能涉及文件修改、目录整理、批量操作或知识库读取
- 用户强调节省 token、减少试错、避免大面积误改
- 智能体需要在 Ask / Plan / Craft / Expert 之间做稳健路由
- 希望在 WorkBuddy、iMA Copilot 等环境中增加安全确认机制

## 不适用场景

不适合用来拖慢本来就很简单的任务。

以下情况不应触发冗长流程：

- 一句话改写
- 简短翻译
- 单条命令解释
- 小模板生成
- 明确、低风险、一次即可完成的小任务

## 四种模式说明

### Ask Mode

适合报错咨询、可行性判断、风险评估、下一步建议、需求模糊时的澄清。默认只诊断、解释、建议，不直接改核心文件。

### Plan Mode

适合复杂任务、多步骤任务、文件整理、知识库处理、批量操作、重构、部署和高风险任务。先出计划，等确认后执行。

### Craft Mode

适合简单明确、低风险、可一次完成的小任务。强调快速交付，不做无意义铺陈。

### Expert Mode

适合多次失败、专业门槛高、继续试错会浪费 token 的情形。它不是假装“专家已经修好”，而是建议调用何种专家视角继续推进。

## 精准聚焦激活机制

这是本次版本新增的关键优化，用来解决“Skill 太多、还没聚焦就开始消耗 token”的问题。

核心规则：

- 每次先选一个主聚焦能力
- 只在必要时再加一个辅助聚焦能力
- 不默认同时展开 Ask / Plan / Craft / Expert / iMA / WorkBuddy 全部分支

推荐聚焦标签：

- `route-only`
- `ask-diagnose`
- `plan-safe-execution`
- `craft-quick-output`
- `expert-escalation`
- `risk-check`
- `need-calibration`
- `knowledge-scope`
- `ima-lite`
- `workbuddy-file`
- `codebuddy-code`

例如：

- 只是问“下一步怎么做”时，不要提前加载 Plan 和 Expert
- 只是 iMA 里做轻量知识判断时，不要提前展开完整报告逻辑
- 只是文件风险判断时，不要默认切入 Craft 生成或大改文件

## 真实需求校准与渐进式推进

这是本次新增的另一层关键机制，用来解决两个常见问题：

1. 用户表面上要求“大批量完整处理”，但真实需求可能只是判断、样本或分阶段推进
2. 智能体还没确认真实需求，就开始消耗大量 token 或进入全量执行

核心逻辑：

- 非必要不消耗
- 先判断真实底层需求
- 先判断是否真的需要一次性完整交付
- 默认优先：`判断 -> 样本 -> 分阶段 -> 一次做完`

也就是说，这个 Skill 不会把“完整做完”当成默认美德，而是把“先用最小投入满足真实需求”当成默认策略。

### 为什么这个逻辑成立

从执行效率看，这个逻辑是成立的，而且通常更高效。

原因是很多大任务的真实目标并不是“把所有东西都做完”，而是：

- 先知道方向对不对
- 先看一个样本是否符合预期
- 先确定执行范围值不值得扩大
- 先避免误改和无效展开

但这个逻辑不能被机械化执行。真正更稳的处理方式是：

- 如果任务低风险、模式稳定、用户明确要求一次完成，那么允许全量交付
- 如果任务范围大、成本高、用途不明，就先做需求校准和渐进推进

这也是本 Skill 现在采用的处理方式。

## 聊天引导按钮与快捷确认

这一版额外补上了“按钮友好型确认”机制，用来解决一个很实际的问题：

- Skill 虽然知道要让用户确认
- 但如果确认文案太长，用户不愿意读，也不愿意回复

所以现在的默认做法是：

- 优先给 `2-4` 个短选项
- 优先给可点、可复制、可直接回复的短标签
- 平台支持按钮时，可直接映射成按钮或快捷回复
- 平台不支持按钮时，也能原样作为文本选项使用

推荐形式：

```text
[1 只要结论]
[2 先做样本]
[3 分阶段]
[4 一次做完]
```

这套形式现在已经接进：

- 交互式路由确认
- 真实需求校准
- iMA 轻量知识库确认
- 文件安全确认
- 继续 / 展开 / 换模式 等后续推进

## 交互式路由确认机制

本 Skill 内置 `Interactive Routing Prompt`。

当任务不够明确，或存在文件、知识库、批量操作、长文、重构、泛化动词等情况时，Skill 不会立刻替用户拍板，而是先给一个极简确认，让用户选择：

1. 任务复杂度
2. 期望模式
3. 输出详细度

支持两种交互方式：

- 完整版路由确认
- 轻量版极简提示

支持快捷编码回复，例如：

- `MPS` = 中等复杂度 + Plan Mode + Standard
- `SPM` = 简单复杂度 + Plan Mode + Micro
- `2B` = 轻量入口中“先给计划 + 标准详细度”

如果用户选择高风险组合，例如“复杂文件任务 + Craft Mode”，Skill 必须拦截并解释为什么需要改用 Plan Mode。

当用户表面上要求“大而全”的结果，但真实需求不明确时，Skill 会优先触发 `真实需求校准`，而不是直接展开完整执行。

## 风险等级说明

### Low Risk

生成文本、小幅润色、单条命令建议、短模板、不改文件的建议。

### Medium Risk

新建文件、少量非核心修改、配置调整、文档整理、多个小文件生成、脚本修改。

### High Risk

删除文件、覆盖文件、批量重命名、批量移动、核心业务文件修改、数据库变更、部署变更、权限调整、自动化任务、长时间运行脚本。

### Critical Risk

读取密钥、读取 cookie、批量上传隐私资料、不可恢复删除、绕过权限、抓取付费内容、执行不明远程脚本、向未知服务器发送用户数据。

## Token 预算模式说明

### Micro

适合极简回复，约 `50-200` 字。常见触发词：

- `简单说`
- `一句话`
- `只给结论`
- `极简`

### Standard

默认模式，约 `300-1500` 字，适合大多数任务。

### Deep

用户明确要求详细分析、系统方案、完整展开时启用，约 `1500-4000` 字。

### Artifact

适合生成文件、模板、Skill、代码、长文档。优先把成果写入文件，聊天中只给摘要、路径和使用说明。

## WorkBuddy 使用建议

WorkBuddy 默认应更保守，因为它经常接触本地文档、知识库、文件夹与自动化任务。

建议规则：

- 默认优先 Ask 或 Plan
- 只有明确低风险任务才直接 Craft
- 涉及文件修改必须 Plan
- 涉及批量操作必须确认
- 涉及知识库整理必须先出计划
- 本地文件任务优先弹出文件安全确认
- 先确认文件范围预算，再决定读多少文件
- 批量任务优先先做一小批样本，不直接全量处理

推荐先给用户这样的选择：

```text
[1 先看 3 个文件]
[2 先看 10 个文件]
[3 先看一个子文件夹]
[4 全目录处理]
```

推荐指令：

```text
请用 token-efficient-task-router 判断这个任务应进入什么模式。
```

```text
请按 Plan Mode 先给我整理方案，不要直接改文件。
```

```text
这个报错先用 Ask Mode 诊断，不要修改代码。
```

```text
这是明确的小任务，请用 Craft Mode 直接生成结果。
```

## iMA Copilot 使用建议

iMA Copilot 主要面向知识库、材料和问答，因此不要默认读取全部资料或直接输出超长综述。

推荐行为：

- 先识别用户到底要查什么
- 先检索最相关的 `3-5` 条材料
- 先输出材料范围
- 先给关键结论
- 需要时再展开
- 先用一句话锁定“模式 + 范围 + 详细度”
- 先选最小聚焦能力，不展开整套 Skill

### iMA 实战优化

为解决 iMA 中常见的两个问题：

1. 对话容易迅速消耗大量 token
2. Skill 规则容易理解不充分、执行漂移

本版新增了三层机制：

- `iMA runtime protocol`
  - 首轮优先控制在一屏内
  - 默认先给判断、范围、结论
  - 支持 `1继续`、`2展开`、`3换模式`、`4只结论`、`5只列材料`
- `focus activation`
  - 先聚焦当前任务只需要的那部分能力
  - 不默认展开全部模式与全部规则
- `need calibration + progressive delivery`
  - 先问用户真正需要判断、样本、分阶段，还是一次做完
  - 不默认把“完整报告”当成必须路径
- `button-ready confirmation`
  - 优先用短选项或快捷回复让用户确认
  - 在支持 UI 的平台中可映射成按钮或 chips

推荐首句：

```text
我先按[模式]处理，只看[范围]，先给你[详细度]结果。
```

知识库任务优先询问：

1. 只看最相关的 3-5 条
2. 先列材料清单
3. 按主题归纳
4. 生成完整报告

默认建议优先选择 `1` 或 `2`。

## CodeBuddy 使用建议

CodeBuddy 里最容易浪费 token 的地方不是输出本身，而是：

- 一上来读太多代码
- 还没定范围就开始改很多文件
- 为了保险直接跑很大的测试范围

建议默认规则：

- 先确认是 `只诊断 / 只读 1-3 文件 / 只改一个点 / 分阶段重构`
- 默认先做最小 diff，不直接大改
- 默认先做最小验证，不直接全量测试
- 整个项目级别的重构必须先做需求校准和阶段规划

推荐先给用户这样的确认：

```text
[1 只诊断]
[2 只读 1-3 文件]
[3 只改一个点]
[4 分阶段重构]
```

如果要验证，再确认：

```text
[1 不跑测试]
[2 最小验证]
[3 相关测试]
[4 完整测试]
```

## 文件安全说明

未经确认，不得：

- 覆盖核心业务文件
- 删除文件
- 批量改名
- 批量移动
- 批量生成复杂目录
- 修改配置文件
- 修改数据库
- 修改自动化脚本
- 修改知识库索引
- 修改发布产物

文件任务开始前，应先说明：

- 会读取哪些文件
- 会修改哪些文件
- 会新建哪些文件
- 不会触碰哪些文件
- 是否建议备份
- 是否需要确认

## 高风险任务确认机制

以下任务必须进入确认闸门：

- 删除或覆盖文件
- 批量重命名或移动
- 核心文件修改
- 数据库、部署、权限、自动化变更
- 大型知识库整理
- 多材料完整报告生成

建议用户用自然语言或快捷码确认，例如：

- `确认执行`
- `只读分析`
- `先生成副本`
- `MPS`
- `2B`

## Expert Mode 召唤机制

当普通模式继续试错已经不划算时，Skill 应明确建议召唤专家，而不是继续高成本试错。

典型专家类型：

- 资深前端工程师
- 资深后端工程师
- 数据分析师
- 表格自动化专家
- 文档结构专家
- 学术论文专家
- 音乐教育研究专家
- 知识库架构专家
- Prompt / Skill 工程师
- 运维部署专家
- 音视频处理专家
- 法律合规顾问
- 财务与成本分析顾问
- 产品经理
- 用户体验设计师
